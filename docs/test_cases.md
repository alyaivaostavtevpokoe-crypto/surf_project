# Тест-кейсы: Ручное тестирование приложения «Картинг-центр» (MVP)

**Роль:** QA Engineer / Systems Analyst  
**Документ:** `docs/test_cases.md`  
**Проверяемый стек:** Flutter (Clean MVVM)  

В данном документе описаны 5 ключевых чек-листов (сценариев) для верификации работы бизнес-логики, экранов расписания и оформления бронирования с учетом заложенных в `MockApiService` ограничений.

---

### Чек-лист №1: Просмотр актуального расписания (Позитивный)
* **Цель:** Убедиться, что список заездов корректно загружается, отображает данные из API и обрабатывает разные типы слотов.
* **Предусловия:** Приложение запущено, `MockApiService` возвращает стандартный набор из 4-х слотов.
* **Шаги тестирования:**
  1. Открыть экран «Расписание заездов».
  2. Дождаться окончания анимации загрузки (`CircularProgressIndicator`).
  3. Проверить визуальное отображение карточки `slot_01` (сегодняшний заезд).
  4. Нажать на кнопку «Обновить» в AppBar.
* **Ожидаемый результат:** - На шаге 2 индикатор пропадает, появляется список карточек.
  - Карточка `slot_01` отображает: дату, интервал времени 12:00 - 12:45, имя инструктора "Алексей Петров", стоимость "2500 ₽", лейбл "Нужен инвентарь" и остаток мест "5/8".
  - На шаге 4 список успешно перезапрашивается без мерцания интерфейса.

---

### Чек-лист №2: Успешное бронирование с арендой экипировки (Позитивный)
* **Цель:** Проверить полный цикл оформления стандартной записи на заезд для одного человека с выбором проката.
* **Предусловия:** На экране отображается список заездов, доступен слот `slot_01` (осталось 5 мест).
* **Шаги тестирования:**
  1. Нажать на карточку заезда `slot_01`.
  2. В открывшейся шторке (`BookingBottomSheet`) проверить, что по умолчанию выбрано 1 место.
  3. Переключить свитчер «Нужна аренда инвентаря» в активное положение.
  4. Проверить, что итоговая сумма равна "2500 ₽".
  5. Нажать на кнопку «Записаться».
  6. После автоматического перехода на `BookingSuccessScreen` нажать кнопку «Вернуться к расписанию».
* **Ожидаемый результат:**
  - На шаге 1 плавно снизу выезжает форма бронирования. Кнопка «Записаться» активна.
  - На шаге 5 на кнопке появляется лоадер на 1 секунду (имитация сети), шторка закрывается.
  - Происходит переход на полноэкранное сообщение об успехе. Шаг 6 корректно возвращает пользователя на главный экран расписания.

### Чек-лист №3: Ограничение вместимости и управление счетчиком мест (Негативный / Граничные значения)
* **Цель:** Проверить, что интерфейс блокирует попытку заказать мест больше, чем физически осталось на данном заезде (`remainingSeats`).
* **Предусловия:** На экране отображается список, доступна карточка `slot_03` (осталось 2 места).
* **Шаги тестирования:**
  1. Нажать на карточку заезда с остатком мест "2".
  2. В шторке бронирования нажать на кнопку "+" один раз (значение увеличится до 2).
  3. Проверить изменение итоговой стоимости (должна стать "6000 ₽").
  4. Нажать на кнопку "+" еще раз.
  5. Нажать на кнопку "-" два раза.
* **Ожидаемый результат:**
  - На шаге 2 счетчик увеличивается, кнопка "+" становится визуально неактивной или нажатие на нее игнорируется, так как достигнут лимит свободных мест заезда (2 из 2).
  - На шаге 4 счетчик остается равным 2. Лимит не превышен.
  - На шаге 5 счетчик не может опуститься ниже 1. Кнопка "-" блокируется при значении 1.

### Чек-лист №4: Отклонение бронирования сервером из-за бизнес-правил (Негативный)
* **Цель:** Проверить обработку системных ошибок бэкенда при попытке обойти лимит группового бронирования (в моке стоит ограничение: не более 4 мест за один раз).
* **Предусловия:** В списке есть заезд с большим количеством мест (например, `slot_04`, где свободно 9 мест).
* **Шаги тестирования:**
  1. Нажать на карточку заезда `slot_04`.
  2. С помощью кнопки "+" увеличить количество мест до 5.
  3. Нажать кнопку «Записаться».
* **Ожидаемый результат:**
  - Кнопка отправки показывает индикатор загрузки.
  - Шторка бронирования **не закрывается**.
  - Внизу экрана всплывает текстовое уведомление (`SnackBar`) красного цвета с понятным текстом ошибки: *"Превышен лимит группового бронирования для одного пользователя (макс. 4 места)."*
  - Интерфейс разблокируется, позволяя пользователю уменьшить число мест.

### Чек-лист №5: Взаимодействие с отмененным заездом (Негативный / Защита от дурака)
* **Цель:** Проверить, что заезды, отмененные мастерской, визуально изолированы и недоступны для проведения каких-либо транзакций.
* **Предусловия:** В списке присутствует карточка `slot_03` со статусом `isCancelledByWorkshop: true`.
* **Шаги тестирования:**
  1. Найти в списке карточку заезда `slot_03`.
  2. Проверить визуальное состояние: текст заголовка должен быть зачеркнут, карточка должна иметь полупрозрачный вид (`opacity: 0.65`).
  3. Проверить наличие блока с причиной отмены.
  4. Попытаться нажать (совершить тап) на эту карточку.
* **Ожидаемый результат:**
  - Карточка окрашена в приглушенные тона, вместо цены отображается серый бейдж «ОТМЕНЕНО».
  - Четко виден красный блок с текстом: *"Причина: Технические работы на покрытии трека (замена отбойников)"*.
  - Нажатие на карточку полностью игнорируется системой. Всплывающая шторка бронирования **не открывается**, защищая пользователя от ошибочной записи.
